TOAST! — FULL PRODUCT REQUIREMENTS DOCUMENT (PRD)
________________


1. PRODUCT OVERVIEW
Product Name
Toast!
Product Type
LINE Mini App (LIFF-based), mobile-first
Core Value Proposition
Toast! helps people decide what to eat faster, especially in groups, by turning food decisions into a simple, social, swipe-based experience.
________________


2. CORE PRINCIPLES (NON-NEGOTIABLE)
1. Decision-first, not discovery-first
2. Group behavior drives growth
3. Friction removal over feature richness
4. Gamification must be subtle and adult
5. Every session must end with a decision or a clear reason why not

________________

3.2 Restaurant Partners
   * Claim and manage restaurant

   * View performance insights
3.3 Admin (Internal)
   * Moderate content

   * Manage ads

   * View stats and data

4. USER APP — FULL FLOW REQUIREMENTS
________________


4.1 Entry & Session Initialization
Entry Points (Similar to Khunthong)
      * Open Toast! via LINE Mini App
      * Open via shared invite link
      * Open via result share link
System Requirements
      * App must load usable UI within <1.5s perceived time
      * Session must restore previous state if user reopens within 10 minutes
________________


4.2 Onboarding (First-Time Users)
Happy Path
      1. Show max 3 onboarding screens
      2. Screens explain:

         * End food debates
         * Decide in under 60 seconds
         * Works best with friends

            3. CTA: “Start Deciding”
Requirements
            * Onboarding is shown once only
            * Must be skippable
            * No account creation required (LINE identity used)
Edge Cases
            * User closes app mid-onboarding → resume next open
            * User opens via group invite → onboarding minimized to 1 screen
________________


4.3 Preference Setup (Optional but Encouraged)
Data Collected
            * Dietary restrictions
            * Allergies
            * Cuisine preferences
            * Budget preference
            * Distance tolerance
Requirements
            * Chip-based selection
            * Skippable
            * Editable later
            * Defaults inferred from behavior if skipped
Edge Cases
            * Conflicting preferences (e.g. vegan + steak)
→ system still proceeds, filtering happens later
            * No preferences selected → neutral weighting
________________


4.4 Mode Selection
Modes
               1. Solo
               2. 1:1
               3. Group
Requirements
               * Group mode visually emphasized
               * Copy: “Best for deciding together”
               * Selecting group mode triggers invite flow
Edge Cases
               * User selects group but does not invite anyone → allow fallback to solo
               * Invite link expires after 24 hours
________________


4.5 Group Creation & Invite Flow
Happy Path
               1. User creates group
               2. System generates:
               * Group ID
               * Invite link
               3. User shares via LINE
Requirements
               * Group must show:
               * Participants
               * Voting status
               * Time estimate
               * Group owner = creator
Edge Cases
               * User joins multiple groups → must select active group
               * User leaves group mid-vote → vote removed
               * Group with only 1 user → prompt to invite others
________________


4.6 Swipe Decision Flow (CORE LOOP)
Card Content
               * Restaurant or dish image
               * Name
               * Distance
               * Price indicator
               * Cuisine tags
Actions
               * Swipe right = Yes
               * Swipe left = No
               * Swipe up = Super Like (Can only use once per group session)
               * Tap = expand details
               * Single Play
               * If Swipe Menu Mode - Click on Menu Card to show list of restaurants that have that specific menu
               * Click restaurant to view details: pictures, address, reviews, deeplink to food delivery apps
               * If Restaurant mode - Click to show details
               * Group play
               * Menu Mode - Once matched will show list of restaurants that has that specific menu
               * Voting to choose the final restaurant
               * Click and view details
               * Restaurant mode - can click to view details
               * Once matched autoshare via group LINE
System Logic
               * Cards filtered based on:
               * Group preferences
               * Availability
               * Distance
               * Other filter options
               * Each swipe recorded per user
Happy Path
               1. User swipes through cards
               2. Votes synced in real-time
               3. System evaluates matches continuously
Edge Cases
               * User disconnects mid-swipe → last confirmed swipe saved
               * User runs out of cards → fallback suggestions shown
               * One user swipes much faster → system waits for others

________________
4.7 Group Lobby & Voting Progress
Requirements
                  * Show:
                  * Avatars
                  * “X votes remaining”
                  * Estimated time to decision
                  * Soft nudges:
                  * Click and sends notification to person in group that hasn’t joined
                  * Countdown expiry triggers:
                  *  Auto-resolution if possible
                  * Removal of inactive users
                  * Inactive users flagged after X minutes
                  * Owner can restart or remove user
Edge Cases
                  * One user idle >5 minutes → mark as inactive
                  * Inactive user removed after 10 minutes (with notice)
________________


4.8 Match Resolution Logic
Match Types
                  * Strong match (majority agrees)
                  * Weak match (simple majority)
                  * Tie / split decision
                  * Display confidence level:
                  * "Strong match (4/5 agree)"
                  * Confidence affects sharing & trust
Resolution Rules
                  * Strong match auto-selected
                  * Tie triggers:
                  * Tie-breaker round OR
                  * Randomized fair selection (configurable)
                  * Voting
                  * Top 3 is only shown when:
                  * No strong match detected
                  * Multiple weak matches exist
                  * Final vote has a single round only
                  * Tie triggers system tie-breaker
                  * No infinite loops allowed
Requirements
                  * Confidence meter displayed
                  * Time-to-decision shown
                  * Decision is final unless group restarts
Edge Cases
                  * No match after all cards → “Expand options” prompt
                  * Users disagree with result → allow restart (counts as new session)
________________


4.9 Result Screen

Requirements

11. SUCCESS CRITERIA
Toast! is successful if:
                           * Median time to decision <90 seconds
                           * 60% sessions include group invites
                           * 40% groups reuse within 7 days
                           * Decisions end arguments, not create them
